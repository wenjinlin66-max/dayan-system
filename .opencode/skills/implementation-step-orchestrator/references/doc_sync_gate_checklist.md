# 文档同步闸门检查单

用于保证“先同步文档，再执行实现”。

## A. 执行前（必须全部为 Yes）

1. 是否已读取 `design-docs/<需求主题>-design-spec.md` 最新版本？
2. 是否已读取实施文档最新版本？
3. 当前计划任务是否全部存在 `Spec-ID -> Task-ID` 映射？
4. 是否确认没有未记录的临时决策？
5. 是否确认设计文档与实施文档不存在冲突描述？
6. （代码类任务）是否已完成前端框架/后端框架/数据库类型判断并记录开发者最终选择？
7. （代码类任务）技术栈决策是否按“每轮一个决策点”的卡片格式记录？

若任一项为 No：停止执行，先更新文档。

## B. 变更触发（任一发生即进入同步流程）

- 需求范围改变
- 设计结构改变
- 方案路径切换
- 接口/数据库/契约调整
- 前端框架/后端框架/数据库类型选择调整
- 实施顺序或验收标准改变

## C. 强制同步流程

1. 更新设计文档（新增/修改条目，必要时更新版本号）
2. 更新实施文档（同步对应步骤、映射、验收）
3. 在双文档写入变更记录（原因、影响、决策人）
4. 重新执行 A 区检查
5. A 区全为 Yes 后，才恢复执行

## D. 禁止行为

- 未同步文档直接改代码或执行新步骤
- 只更新其中一份文档
- 有口头/对话变更但无文档落地
- 跳过映射关系直接新增任务

## E. 最终放行条件

- 双文档版本一致
- 关键术语定义一致
- `Spec-ID -> Task-ID` 可追溯
- 变更记录完整
- 当轮可执行任务与文档内容 100% 一致
