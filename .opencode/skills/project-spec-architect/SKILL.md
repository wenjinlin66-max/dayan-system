# 项目规格架构师

## 概述

该 skill 面向软件/网页项目的 0 到 1 规划阶段，帮助用户把想法转成可执行文档与开发蓝图。
核心目标是：先定义“做什么”，再明确“怎么做”，最后锁定“如何验收”。

## 触发条件

当用户出现以下意图时优先使用本 skill：
- 写需求文档、PRD、功能清单
- 设计系统架构、项目目录、模块边界
- 定义前后端接口、字段、错误码、契约
- 规划迭代路线、里程碑、验收标准

示例触发语句：
- “帮我做这个项目的需求文档”
- “先把项目框架和接口定义出来”
- “给我一份 API 契约和开发计划”

## 工作流（按顺序执行）

### 第 1 步：需求澄清与范围冻结
- 明确目标用户、核心场景、业务边界、非目标
- 梳理功能优先级（P0/P1/P2）
- 输出：需求摘要 + 范围边界

### 第 2 步：产出 PRD
- 按模板生成：背景、目标、用户故事、功能需求、非功能需求、验收标准
- 每个功能必须有“输入-处理-输出-异常”
- 输出：`references/prd_template.md` 对应结构的 PRD 初稿

### 第 3 步：设计系统架构与项目框架
- 划分模块边界（前端、后端、数据层、外部依赖）
- 明确目录结构、关键组件、依赖关系
- 输出：架构说明 + 建议目录树

### 第 4 步：定义接口与契约
- 统一接口风格：路径、方法、鉴权、请求体、响应体、错误码
- 明确字段类型、可空性、默认值、枚举范围
- 输出：接口清单与契约文档（可用于前后端并行开发）

### 第 5 步：实施计划与验收策略
- 迭代拆分（M1/M2/M3），每期给出可交付物
- 测试策略：单元、集成、回归、接口契约测试
- 输出：实施计划、风险清单、验收清单

## 输出规范

每次使用本 skill，至少输出以下 4 块内容：
1. PRD（精简或完整版）
2. 架构与项目框架建议
3. API 契约（含错误码）
4. 实施计划与验收标准

推荐输出顺序：先业务目标，再架构，再接口，再排期。

## 质量检查清单

- 需求是否可测试、可验收
- 架构是否与需求范围一致，避免过度设计
- 接口是否字段定义完整且前后一致
- 是否给出失败场景与错误码
- 是否有明确里程碑和验收口径
