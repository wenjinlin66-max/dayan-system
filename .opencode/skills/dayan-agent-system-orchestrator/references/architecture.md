# 架构设计模板

## 1. 总体拓扑

## 2. 前端工作台体系
### 2.1 架构师画布工作台
- 只需覆盖智能体协同模块相关页面：工作流画布、节点配置、草稿编译发布、版本查看
### 2.2 部门员工对话工作台
- 只需覆盖智能体协同模块相关页面：会话列表、对话窗口、审批卡片、执行结果回传
### 2.3 运维总管监控工作台
- 只需覆盖智能体协同模块相关页面：执行列表、异常任务、指标卡片、死信查看
### 2.4 业务表格工作台
- 用于展示和编辑独立 Mock 业务数据库 `dayan_mock_records`
- 页面目标不是替代画布、对话、监控，而是提供“真实业务表 -> 感知触发 -> workflow 执行 -> 结果写回”的联调入口
- 应作为一级页面与其他工作区并列存在
- 该工作台定位为**临时测试工作区**；Go 正式表格/records 能力接入后，应整体删除，不作为正式产品长期页面保留

### 2.5 顶部导航与工作流查看区
- 前端顶层信息架构当前已收口为统一顶部导航栏
- 顶部导航至少覆盖：`工作流制作区 / 工作流查看区 / 对话区 / 监控区 / 业务表格区`
- `工作流查看区` 用于展示当前账号所在部门有权限查看的所有 workflow，后端必须做 `dept_id` 硬限制，不再依赖前端部门筛选兜底
- `工作流查看区` 的正式分类逻辑收口为：未来按“部门 → 触发逻辑”两层组织；在部门口径尚未单独确定前，前端先按触发逻辑分类展示
- 当前触发逻辑分类至少预留：`定时触发 / 事件触发（感知触发）/ 对话触发（对话框触发）`
- 用户不再依赖画布页中的临时入口跳转到对话区，而是通过统一导航在各工作区之间切换
- `业务表格区` 当前推荐放在 `监控区` 旁边，作为新的顶层导航项
- 该导航项当前仅服务于未接 Go 正式表格前的测试联调；Go 侧正式接管后，允许从顶部导航整体移除

## 3. 后端分工
### 3.1 Go 数据底座
 - 负责企业业务主数据、泛型 API、权限主权、审批主记录、业务事件抛出
 - 不负责工作流逻辑 JSON 的存储与解析
### 3.2 Python AI 中枢
 - 负责 workflow definitions / versions / execution_dag 的真相存储
 - 负责工作流草稿保存、编译、发布、执行、恢复
 - 负责将画布编辑态编译为执行态逻辑 JSON
- 在未接 Go 正式业务库前，负责通过独立数据库 `dayan_mock_records` 提供 Mock Records 联调层，但该联调层不得与主库 `dayan_agentic2` 混存
- 该联调层属于过渡测试设施，不应演化为长期正式数据底座

## 4. 智能体体系
### 4.1 感知型智能体
- 负责五类外部/内部信号进入工作流：
  1. IoT 设备数据采集与状态上报
  2. 表单数据变更事件捕获
  3. 供应链异常事件监听
  4. 第三方系统通知接收
  5. 定时数据巡检
- 前端可配置：监控对象、筛选条件、触发条件、事件映射方式
- 输出：标准化事件信封，作为工作流初始输入

#### 4.1.1 当前阶段实现边界（MVP）
- 第一阶段只正式实现：**数据库实时感知**
- 具体含义：Go 后端业务数据发生变更后，通过事件总线将变更事件推送给 Python；Python 的感知型节点根据前端配置判断是否命中，并向下游工作流投递标准化事件
- 第一阶段不立即实现但必须预留结构：IoT / 第三方 Webhook / 供应链异常回调 / 定时巡检

#### 4.1.2 数据库实时感知主链
1. Go 业务层更新数据库记录
2. Go 通过 Asynq / Redis 推送标准化数据变更事件
3. Python 感知型智能体消费事件
4. 按节点配置完成：
   - 数据源匹配
   - 字段过滤
   - 条件判断
   - 事件字段映射
5. 命中后生成统一 `sensor_event`
6. 投递到对应 workflow execution

#### 4.1.2.1 当前无 Go 正式库时的替代主链
1. 用户进入业务表格工作台查看 `dayan_mock_records`
2. 前端通过 Python Mock Records API 修改某条业务记录
3. Python 将该修改写入独立数据库 `dayan_mock_records`
4. Python `event bridge` 把此次变更包装成标准数据库事件
5. `sensor_agent` 按配置做来源匹配、字段过滤、条件判断、输出映射
6. 命中后触发 workflow execution
7. `execution_agent` 可继续把结果写回 `dayan_mock_records`
8. 前端业务表格工作台刷新后看到真实修改结果

#### 4.1.3 感知型节点前端配置建议
前端感知型节点配置采用“分区引导式”结构：
- 区域 1：基础身份
  - 启用状态
  - 所属部门
  - 环境
  - 节点名称
- 区域 2：数据选择（连接器 → 表 → 字段）
  - 数据源系统
  - 表/资源选择
  - 事件源标识
  - 关注字段勾选
- 区域 3：逻辑构建器
  - 条件组合（AND / OR）
  - 字段、操作符、比较值
  - JSON 预览
- 区域 4：事件输出映射
  - 输出事件名称
  - 输出字段映射
  - 是否透传原始 payload

#### 4.1.4 后续预留扩展
- IoT：通过 MQTT / 工业协议网关适配为统一事件源
- Webhook：通过 HTTP 接入第三方系统通知
- 供应链异常：接外部系统回调或异步事件
- 定时巡检：由 Python 调度器主动拉取数据并生成事件
### 4.2 决策型智能体
- 支持三种决策模式：规则型、模型型、智能型
- 智能型默认具备三层记忆接口：上下文记忆、历史执行记忆、RAG 知识库记忆
- 输出：结构化决策结果、理由、引用来源、风险等级、建议动作

#### 4.2.1 三种决策模式定义
1. 规则型决策（Rule）
   - 基于预设规则、公式、条件判断
   - 适用于标准化、固定规则明确的场景
   - 特点：可解释、稳定、速度快

2. 模型型决策（Model）
   - 基于运筹优化、预测模型、机器学习模型
   - 适用于排程、优化、预测等复杂计算场景
   - 特点：强调参数化、可调目标、约束求解

3. 智能型决策（LLM + RAG）
   - 基于大语言模型 + 知识检索
   - 适用于异常处理、风险判定、复杂场景综合决策
   - 特点：上下文理解强，能结合历史执行与知识库输出解释

#### 4.2.2 决策型节点前端配置建议
前端决策型节点配置采用“模式选择 + 模式专属配置 + 统一输出配置”结构：
- 区域 1：基础身份
  - 启用状态
  - 所属部门
  - 环境
  - 节点名称
- 区域 2：模式选择
  - 规则型 / 模型型 / 智能型 单选
- 区域 3：模式专属配置
  - 规则型：规则集、公式、阈值、条件表达式
  - 模型型：模型类型、参数、优化目标、约束条件
  - 智能型：提示词模板、记忆开关、RAG 知识源、输出格式
- 区域 4：输出配置
  - 输出事件名
  - 输出字段映射
  - 是否附带解释理由 / 引用来源

#### 4.2.3 决策输出统一口径
无论决策型节点使用规则型、模型型还是智能型，输出都必须统一为结构化决策结果，至少包含：
- `decision_mode`
- `decision_summary`
- `decision_payload`
- `risk_level`
- `recommended_actions`
- `explanation`
- `citations`

这样可以保证下游执行型、审批型、对话型节点不需要分别适配三套输出结构。
### 4.3 执行型智能体
- 支持 MCP / API / 第三方工具调用
- 支持新增抽象执行对象：`department_table`，用于按部门将结构化数据写入各部门对应表格
- 可由 AI 根据执行方案和上下文自主选择工具
- 高风险动作支持转审批节点，由对话界面人工确认后继续执行
- 输出：执行结果、执行日志、工具调用轨迹、失败重试信息

#### 4.3.1 当前阶段实现边界（MVP）
- 第一阶段优先实现：**对话框审批主链 + 执行结果回传对话框**
- 第一阶段补充纳入：**部门表格写入执行对象**，作为执行型智能体首个通用业务落地目标之一
- 第一阶段暂不优先实现：飞书、邮件、外部设备、更多第三方工具真实接入
- 但必须预留后续扩展位：工具注册表、执行目标适配器、第三方执行器

#### 4.3.2 执行型节点的两种执行目标模式
1. 手动选择执行目标
   - 用户/配置员在设置面板明确选择执行目标
   - 如：Go 泛型 API、部门表格、飞书、邮件、设备控制、文件处理等

2. AI 判断执行目标
   - 用户/配置员只给执行目标范围或策略
    - 执行型智能体根据决策结果、上下文和工具注册表选择最合适的执行目标

当前阶段补充执行原则：
- 一个 `execution_agent` 节点默认负责一个主执行目标，例如：`department_chat` / `department_table` / `feishu` / `email` / `mcp`
- 如果同一条业务决策需要“发对话框报告 + 修改业务表格 + 调第三方工具”等多个动作，应通过 `parallel` 控制节点把多个 `execution_agent` 并列挂在 `decision_agent` 后，而不是把所有目标挤在同一个 execution 节点里混做
- `execution_target_mode=ai_select` 只解决“当前 execution 节点在候选目标中选一个”的问题；若业务上明确要多个动作一起做，仍应采用多 execution 节点并列

#### 4.3.3 部门表格执行对象
- `department_table` 是执行型智能体的标准执行目标之一，不绑定单一厂商
- 它的目标是：根据 `dept_id`、节点配置和字段映射，将结构化业务数据写入该部门对应表格
- 表格可承载于不同实现载体，如在线多维表、共享表单表格、轻量业务台账等，但在执行型智能体中统一抽象为同一种目标类型

执行原则：
1. 执行型智能体先从上下文获取 `dept_id`
2. 再从 `execution_target_registry` 中查找该部门可用的 `department_table` 目标
3. 根据节点配置完成字段映射、默认值补齐、幂等键生成、写入策略判定
4. 如命中审批策略，则先进入对话审批，再执行写入
5. 写入后将结果摘要、表格目标、写入行标识回传到对话工作台与审计日志

最小能力要求：
- 支持按部门路由到不同表格
- 支持 `append_row` 为第一阶段默认动作
- 预留 `upsert_row` / `update_row` 扩展位
- 支持字段映射、固定值、上下文字段透传
- 支持幂等写入与重复提交防护
- 支持失败回传与重试摘要

#### 4.3.4 审批与对话主链
1. 决策型智能体输出统一结构化 `recommended_actions`
2. 执行型智能体根据配置决定：
    - 直接执行
    - 进入审批
3. 若进入审批：
   - 生成审批卡片
   - 推送到对话工作台的审批工作区
4. 用户在对话框完成同意 / 驳回
5. 执行型智能体根据审批结果继续执行或终止
6. 执行结果、失败信息、工具日志摘要回传到对话工作台

当前阶段补充口径：
- 审批不再只理解为独立 `approval` 控制节点；`execution_agent` 本身也可以在命中 `approval_required + approval_mode` 后先挂起，再由审批工作区恢复继续执行当前节点
- `department_chat` 目标当前表示“把决策型结果整理为风险报告并发送到目标部门对话框”；报告格式应优先由决策型输出口径 + execution 模板共同决定
- `department_table` 目标当前继续承担业务表格写入；若同时还要发送正式风险报告到对话区，推荐由 parallel 分支中的另一个 `department_chat` 执行节点承担，而不是把 chat 当作表格写入后的附赠副作用

#### 4.3.5 对话工作台新增要求
- 必须存在独立“审批工作区”
- 审批工作区至少展示：
  - 待审批方案摘要
  - 风险等级
  - 推荐动作
  - 同意 / 驳回按钮
  - 审批意见输入
- 执行完成后，对话流中可见：
  - 执行结果摘要
  - 执行状态
  - 失败原因（如失败）
  - 若执行目标为 `department_table`，需展示目标表格标识、写入行结果或失败位置
- 审批挂起与审批结果都应回写到当前部门对话框消息流，而不是只停留在侧边待办区
- 执行结果回传默认进入当前触发它的部门对话框，保证用户在同一对话上下文中看到审批与结果闭环
- 对话区在视觉与交互上应优先表现为“AI 问答主窗口”，审批、流程目录、执行结果为围绕主问答的辅助能力，而不是平铺同级区域
### 4.4 对话型智能体
- 作为用户唯一入口，负责接收提问、审批、命令执行
- 支持文本、语音、PDF 等多模态输入
- 可查询部门信息、调用数据库、检索 RAG 知识库、驱动授权工作流执行
- 对不同部门用户施加不同数据权限与工作流使用边界

#### 4.4.1 对话型智能体的三条主路径
对话型智能体必须先将用户输入路由到三类路径之一：
1. `ask`：部门问答 / 数据查询 / 知识检索
2. `approve`：审批任务处理
3. `command`：工作流启动或执行命令

#### 4.4.2 部门化主入口原则
- 每个部门登录后，只能看到本部门工作流、审批任务、业务信息与知识范围
- 同部门账号权限可相等，但操作记录、聊天记录、审批记录必须按用户分别留痕
- 对话型智能体是部门用户的唯一操作入口，用户不直接从画布启动工作流
- 对话工作台的组织方式以“部门对话框 / 部门会话”为主，不按单条 workflow 拆独立入口；workflow 只是部门对话框中的可执行对象
- 默认问答模型允许通过 OpenAI-compatible 中转站接入 Gemini 类模型，但密钥必须仅通过环境变量注入，不能写入 skill 文档与代码

#### 4.4.3 工作流发现与路由原则
- 对话型智能体不能仅靠自由语言直接硬选 workflow
- 画布发布后的 workflow 必须进入“workflow 目录/注册表”
- 对话型智能体先从目录中检索候选 workflow，再结合部门权限、角色、必填参数、风险策略进行过滤
- 若候选冲突或语义歧义明显，不得自动启动，必须在对话框中让用户二次确认

#### 4.4.3.1 对话型智能体选择 workflow 的技术主链
对话型智能体选择 workflow 不采用“单次大模型硬判断”，而采用“语义理解 + 目录检索 + 规则过滤 + 必要时确认”的组合模式：
1. 先将用户输入分类到 `ask / approve / command` 三类路径
2. 若为 `command`，先抽取部门、场景关键词、已给出的业务参数
3. 再到 `workflow_registry` 中检索候选 workflow，检索依据包括：
   - title
   - summary
   - synonyms
   - example_utterances
   - category
4. 检索到候选后，再按以下规则收缩范围：
   - `dept_scope`
   - `allowed_roles`
   - `required_inputs`
   - `risk_level`
   - `approval_policy`
5. 若高置信度仅剩一条，允许进入启动准备
6. 若存在多条候选或语义冲突，则必须在对话框中让用户确认，而不是自动硬选

结论：
- 大模型负责理解“用户想办什么事”
- workflow registry 负责提供“系统里有哪些流程可选”
- 规则过滤负责保证“不越权、不误选、不漏参数”

#### 4.4.3.2 对话型智能体触发 workflow 的技术主链
对话型智能体本身不直接执行业务流程，而是负责完成启动前准备并调用 Python workflow 执行入口：
1. 完成 workflow 选择
2. 完成必填参数抽取；若参数不足，则继续对话补齐
3. 生成标准启动上下文：
   - `workflow_id`
   - `version`
   - `mode`
   - `dept_id`
   - `operator`
   - `trigger.type=chat`
   - `session_id / message_id`
   - `input`
4. 将上述上下文提交给 Python execution 启动入口
5. Python workflow 引擎读取已发布 DAG，从入口节点开始执行
6. 执行状态、审批卡片、最终结果再回流到对话工作台

结论：
- 对话框控制的不是“节点级执行细节”
- 对话框控制的是“选择哪条 workflow、何时启动、带什么参数启动、何时恢复执行”
- 真正的 workflow 运行仍由 Python 执行中枢负责

#### 4.4.4 workflow 目录/注册表最小信息
每个已发布 workflow 至少应提供：
- workflow_id
- version
- title
- summary
- synonyms
- example_utterances
- dept_scope
- allowed_roles
- required_inputs
- input_schema
- approval_policy
- risk_level
- output_contract

为了支撑对话型智能体稳定选流，推荐额外补充：
- category
- start_mode（如 `chat | event | manual`）
- confirmation_required
- confidence_hint

#### 4.4.5 审批工作区
- 对话工作台必须包含独立审批工作区
- 审批工作区与 chat 流共享同一任务对象，不允许各自维护两套审批状态
- 审批卡片至少展示：
  - 方案摘要
  - 风险等级
  - 推荐动作
  - 审批意见输入
  - 同意 / 驳回

### 4.5 监控型智能体
- 提供主动与被动巡查能力
- 负责查看执行结果、异常状态、超时、失败率、死信与恢复情况
- 面向监控工作台输出可观测数据

#### 4.5.1 定位原则
- 监控型智能体不是普通 workflow 画布节点
- 它应作为**独立控制平面（control-plane）子系统**存在
- 它不参与业务数据处理，不承担业务节点逻辑，只负责观察、评估、告警与有限干预

#### 4.5.2 核心职责
监控型智能体负责 5 类监控规则族：
1. 流程超时监控与预警
2. 数据一致性校验
3. Agent 执行异常告警与兜底建议
4. 业务合规性校验
5. 系统健康状态巡检

#### 4.5.3 数据来源
监控型智能体的观察数据应来自控制面数据，而不是临时内存状态：
- execution 状态事件
- step status / heartbeat
- audit logs
- retry history
- operator actions
- queue/backlog metrics
- worker health

#### 4.5.4 有限干预动作
监控型智能体可以通过显式运行时 API 发起受控动作，但不能成为隐藏执行引擎：
- `pauseExecution`
- `retryStep`
- `markForReview`
- `quarantineOutput`
- `resumeExecution`

所有干预动作必须：
- 有审计记录
- 有权限校验
- 可追溯发起人 / 发起规则 / 触发原因

#### 4.5.5 告警模型
- 监控型智能体不直接生成一次性零散告警
- 推荐先将原始监控信号聚合为 `incident`
- incident 至少包含：
  - severity
  - scope
  - owner
  - dedup_key
  - escalation_timer
  - resolution_state

#### 4.5.6 为什么不上画布
- 监控型智能体是跨流程、跨模块、跨节点的监督能力
- 若做成画布节点，会误导为“某条流程上的可选装饰”
- 即使 workflow 本身出错或未发布，监控型智能体仍应独立可运行

## 5. 控制节点体系
- 条件分支
- 并行
- 循环
- 子流程
- 等待
- 审批
- 异常兜底

### 5.1 控制节点定位
控制节点不直接调用大模型，不直接操作业务数据库，它们只负责决定 workflow 的下一步流向、节奏与边界。

### 5.2 七类控制节点定义

#### 5.2.1 条件分支节点（Branching）
- 用途：实现 `if-else` / `switch-case` 路由
- 输入：上游节点输出或上下文状态
- 输出：命中某一个目标分支

#### 5.2.2 并行网关节点（Parallel）
- 用途：同时启动多个分支任务（fork）
- 输入：统一父上下文
- 输出：多分支并行执行结果，必要时 join 汇总

#### 5.2.3 循环执行节点（Loop）
- 用途：定义循环次数或循环条件
- 输入：循环体节点与退出条件
- 输出：直到满足退出条件或达到最大次数

#### 5.2.4 子流程节点（Sub-flow）
- 用途：嵌套另一个 workflow
- 输入：子流程 ID / version + 输入映射
- 输出：子流程执行结果

#### 5.2.5 等待节点（Wait）
- 用途：阻塞一段时间，或等待外部事件/信号
- 输入：时间、外部事件键、恢复条件
- 输出：恢复后继续流转

#### 5.2.6 人工审批节点（Approval）
- 用途：挂起流程，等待用户审批信号
- 输入：审批卡片配置、审批人范围、恢复映射
- 输出：同意 / 驳回 / 超时等结果

#### 5.2.7 异常兜底节点（Exception）
- 用途：当上游报错时自动跳转到此节点进行补救
- 输入：错误上下文、失败节点信息、补偿策略
- 输出：补救结果或终止结果

### 5.3 当前阶段边界
- 当前阶段先完成控制节点定义、DSL 字段、前端配置与运行时边界设计
- 后续阶段再逐步实现具体执行逻辑
- 审批节点因已与执行型/对话型主链耦合，可优先实现

## 5.1 工作流真相与发布机制
- 画布编辑态（ui_schema）不等于执行态（execution_dag）
- 工作流逻辑 JSON、版本、发布状态由 Python 持有
- Python 直接执行已发布版本；如开启沙盒，可执行草稿测试版
- Go 不发布 workflow，只消费或配合 Python 暴露的执行结果、审批与业务事件
- 前端点击发布时，应触发 Python 的编译与发布流程，而不是依赖 Go 中转

## 7.1 用户使用工作流的三种模式

## 7.2 工作流使用规则

- 工作流必须显式绑定授权部门，未授权部门不可调用
